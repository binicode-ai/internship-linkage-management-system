# Requirements Document

## Introduction

This document defines the requirements for the Internship & Industry Linkage Management System — a full-stack, database-driven web application built with Node.js, Express.js, PostgreSQL, and EJS templating. The system digitalizes and streamlines the complete internship lifecycle within the Ethiopian university context, connecting universities, students, and companies. It extends an existing partial implementation to cover university management, company onboarding, internship opportunity management, student applications, supervisor assignment, recommendation letter generation, and performance evaluation.

## Glossary

- **System**: The Internship & Industry Linkage Management System web application
- **University**: An Ethiopian higher education institution registered in the system
- **Student**: A university-enrolled individual seeking an internship placement
- **Company**: An industry organization registered to offer internship opportunities
- **Internship**: A time-bound work placement opportunity posted by a Company
- **Application**: A Student's formal request to be considered for an Internship
- **University_Supervisor**: A university staff member assigned to supervise a Student during an Internship
- **Company_Supervisor**: A company employee assigned to supervise and evaluate an intern
- **Evaluation**: A structured performance assessment submitted by a Company_Supervisor for an intern
- **Recommendation_Letter**: An official document generated by the System endorsing a Student for an Internship
- **Dashboard**: The main navigation page of the System
- **DB**: The PostgreSQL relational database backing the System

## Requirements

### Requirement 1: University Management

**User Story:** As an administrator, I want to register and manage universities, so that students can be associated with their respective institutions.

#### Acceptance Criteria

1. THE System SHALL provide a form to register a new University with fields: university name, location, and contact email.
2. WHEN a university registration form is submitted with all required fields, THE System SHALL insert the University record into the DB and redirect to the universities list page.
3. IF a university registration form is submitted with a duplicate university name, THEN THE System SHALL return a descriptive error message without inserting a duplicate record.
4. THE System SHALL display a list of all registered Universities showing university name, location, and contact email.
5. WHEN a University record is deleted, THE System SHALL prevent deletion if Students are still associated with that University and display an explanatory error message.

---

### Requirement 2: Student Registration and Management

**User Story:** As a university administrator, I want to register students and associate them with their university, so that students can apply for internships.

#### Acceptance Criteria

1. THE System SHALL provide a form to register a new Student with fields: first name, last name, email, department, year of study, and university (selected from registered Universities).
2. WHEN a student registration form is submitted with all required fields, THE System SHALL insert the Student record into the DB and redirect to the students list page.
3. IF a student registration form is submitted with an email address already present in the DB, THEN THE System SHALL return a descriptive error message without inserting a duplicate record.
4. THE System SHALL display a list of all registered Students showing first name, last name, email, department, year of study, and affiliated university name.
5. WHEN a Student record is viewed, THE System SHALL display all Applications submitted by that Student along with their current status.

---

### Requirement 3: Company Registration and Management

**User Story:** As an administrator, I want to register and manage company profiles, so that companies can post internship opportunities.

#### Acceptance Criteria

1. THE System SHALL provide a form to register a new Company with fields: company name, industry type, location, contact email, and phone number.
2. WHEN a company registration form is submitted with all required fields, THE System SHALL insert the Company record into the DB and redirect to the companies list page.
3. IF a company registration form is submitted with a duplicate contact email, THEN THE System SHALL return a descriptive error message without inserting a duplicate record.
4. THE System SHALL display a list of all registered Companies showing company name, industry type, location, contact email, and phone number.
5. WHEN a Company record is deleted, THE System SHALL prevent deletion if active Internships are associated with that Company and display an explanatory error message.

---

### Requirement 4: Internship Opportunity Management

**User Story:** As a company representative, I want to post and manage internship opportunities, so that students can discover and apply for them.

#### Acceptance Criteria

1. THE System SHALL provide a form to post a new Internship with fields: company (selected from registered Companies), title, description, duration in months, start date, and stipend amount.
2. WHEN an internship posting form is submitted with all required fields, THE System SHALL insert the Internship record into the DB and redirect to the internships list page.
3. THE System SHALL display a list of all posted Internships showing title, company name, duration, start date, stipend, and current status.
4. WHEN an Internship's start date has passed and no Application has been accepted for it, THE System SHALL display the Internship with a status of "Open".
5. WHEN at least one Application for an Internship has been accepted, THE System SHALL display the Internship with a status of "Filled".
6. THE System SHALL allow an Internship record to be updated to change description, duration, start date, or stipend.

---

### Requirement 5: Student Application Submission and Tracking

**User Story:** As a student, I want to submit and track my internship applications, so that I can monitor the progress of my placement process.

#### Acceptance Criteria

1. THE System SHALL provide a form for a Student to apply for an Internship by selecting a student and an internship from dropdown lists populated from the DB.
2. WHEN an application form is submitted, THE System SHALL insert the Application record into the DB with a default status of "Pending" and redirect to the applications list page.
3. IF a Student attempts to submit a duplicate Application for the same Internship, THEN THE System SHALL return a descriptive error message without inserting a duplicate record.
4. THE System SHALL display a list of all Applications showing student full name, internship title, company name, application date, and current status.
5. WHEN an administrator updates an Application status to "Accepted", THE System SHALL update the Application record in the DB and make the associated Internship available for supervisor assignment.
6. WHEN an administrator updates an Application status to "Rejected", THE System SHALL update the Application record in the DB.
7. THE System SHALL allow filtering of the applications list by status (Pending, Accepted, Rejected).

---

### Requirement 6: Supervisor Assignment

**User Story:** As an administrator, I want to assign supervisors from both universities and companies to accepted interns, so that students receive proper guidance throughout their internship.

#### Acceptance Criteria

1. THE System SHALL provide a form to assign a University_Supervisor to an accepted Application, with fields: supervisor full name, email, and affiliated university.
2. THE System SHALL provide a form to assign a Company_Supervisor to an accepted Application, with fields: supervisor full name, email, job title, and affiliated company.
3. WHEN a supervisor assignment form is submitted with all required fields, THE System SHALL insert the supervisor assignment record into the DB and confirm the assignment.
4. IF a supervisor assignment form is submitted for an Application that does not have "Accepted" status, THEN THE System SHALL return a descriptive error message and reject the assignment.
5. THE System SHALL display a list of all supervisor assignments showing intern name, internship title, university supervisor name, and company supervisor name.
6. WHEN a supervisor is reassigned for an Application, THE System SHALL update the existing supervisor assignment record rather than inserting a duplicate.

---

### Requirement 7: Recommendation Letter Generation

**User Story:** As a university administrator, I want to generate official internship recommendation letters for accepted students, so that students have formal documentation for their placement.

#### Acceptance Criteria

1. WHEN an Application status is set to "Accepted", THE System SHALL make a "Generate Recommendation Letter" action available for that Application.
2. WHEN a recommendation letter is generated, THE System SHALL produce a letter containing: student full name, university name, internship title, company name, start date, and duration.
3. THE System SHALL store a record of each generated Recommendation_Letter in the DB, including the generation timestamp and associated application ID.
4. THE System SHALL render the Recommendation_Letter as a printable HTML page accessible via a unique URL.
5. IF a Recommendation_Letter has already been generated for an Application, THEN THE System SHALL retrieve and display the existing letter rather than generating a duplicate.

---

### Requirement 8: Internship Evaluation Submission

**User Story:** As a company supervisor, I want to submit performance evaluations for interns, so that universities and students receive formal feedback on internship performance.

#### Acceptance Criteria

1. THE System SHALL provide a form for a Company_Supervisor to submit an Evaluation for an intern, with fields: application ID, technical skills score (1–5), communication score (1–5), teamwork score (1–5), overall score (1–5), and comments.
2. WHEN an evaluation form is submitted with all required fields and valid score ranges, THE System SHALL insert the Evaluation record into the DB and redirect to the evaluations list page.
3. IF an evaluation form is submitted with a score outside the range of 1 to 5 for any metric, THEN THE System SHALL return a descriptive validation error without inserting the record.
4. IF an evaluation form is submitted for an Application that does not have "Accepted" status, THEN THE System SHALL return a descriptive error message and reject the submission.
5. IF an Evaluation already exists for an Application, THEN THE System SHALL allow the Company_Supervisor to update the existing Evaluation record rather than inserting a duplicate.

---

### Requirement 9: Evaluation Results Retrieval and Display

**User Story:** As a university administrator, I want to view and retrieve internship evaluation results, so that I can monitor student performance and maintain academic records.

#### Acceptance Criteria

1. THE System SHALL display a list of all submitted Evaluations showing intern full name, company name, technical skills score, communication score, teamwork score, overall score, and comments.
2. WHEN an Evaluation record is viewed in detail, THE System SHALL display all performance metrics alongside the intern's name, university, company, and internship title.
3. THE System SHALL allow filtering of the evaluations list by company name or by university name.
4. THE System SHALL calculate and display the average score across all four metrics for each Evaluation record.
5. WHILE an Evaluation record exists in the DB, THE System SHALL make it retrievable by application ID via a dedicated route.

---

### Requirement 10: Database Schema and Integrity

**User Story:** As a developer, I want a normalized relational database schema, so that data integrity is maintained and the system is free of redundancy.

#### Acceptance Criteria

1. THE DB SHALL contain tables for: universities, students, companies, internships, applications, university_supervisors, company_supervisors, recommendation_letters, and evaluations.
2. THE DB SHALL enforce primary key constraints on every table.
3. THE DB SHALL enforce foreign key constraints between all related tables to maintain referential integrity.
4. THE DB schema SHALL be normalized to at least Third Normal Form (3NF), with no transitive dependencies between non-key attributes.
5. WHEN a parent record is deleted and dependent child records exist, THE DB SHALL reject the deletion and return a foreign key constraint violation.
6. THE DB SHALL enforce a unique constraint on student email addresses and company contact email addresses.

---

### Requirement 11: Navigation and User Interface

**User Story:** As a user, I want a consistent and navigable web interface, so that I can access all system features without confusion.

#### Acceptance Criteria

1. THE System SHALL render a Dashboard page at the root URL ("/") that provides navigation links to all major sections: Universities, Students, Companies, Internships, Applications, Supervisors, Recommendation Letters, and Evaluations.
2. THE System SHALL include a consistent header and footer partial on all EJS views.
3. WHEN a form submission results in an error, THE System SHALL re-render the form page with the error message displayed inline, preserving previously entered field values where possible.
4. WHEN a form submission is successful, THE System SHALL redirect to the relevant list page rather than displaying a plain text confirmation message.
5. THE System SHALL serve static CSS styles from the public directory to provide consistent visual formatting across all pages.

---

### Requirement 12: Error Handling and Resilience

**User Story:** As a user, I want the system to handle errors gracefully, so that unexpected failures do not expose raw error details or crash the server.

#### Acceptance Criteria

1. WHEN a database query fails due to a connection error, THE System SHALL log the error to the server console and render a user-friendly error page with a 500 HTTP status code.
2. WHEN a request is made to a route that does not exist, THE System SHALL render a 404 error page.
3. IF required form fields are missing from a POST request body, THEN THE System SHALL return a 400 HTTP status code with a descriptive error message.
4. WHILE the System is running, THE System SHALL not expose raw PostgreSQL error messages or stack traces to the end user in the browser.
